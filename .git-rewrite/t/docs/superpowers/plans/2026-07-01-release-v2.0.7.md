# CarExplorer v2.0.7 Release Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Release CarExplorer v2.0.7 with premium ad-free features, release automation infrastructure, and comprehensive test coverage.

**Architecture:** The release process follows a standard mobile app workflow: (1) verify build readiness via test suite, (2) bump version numbers for semantic versioning and Play Store tracking, (3) merge feature branch to main via PR, (4) create git release tag to trigger automated build workflow, (5) generate release notes documenting changes, (6) monitor build workflow completion. All build and Play Store upload steps are automated via GitHub Actions workflows already in place.

**Tech Stack:** 
- React Native + Expo for mobile app
- GitHub Actions for CI/CD automation
- Gradle for Android build (AAB bundle format)
- Play Store for distribution

## Global Constraints

- Version format: Semantic versioning (MAJOR.MINOR.PATCH) in `app.json` → `version` field
- versionCode: Android build number, must increment by 1 for each release in `app.json` → `android.versionCode`
- Git tag format: `v<version>` (e.g., `v2.0.7`) to trigger build workflow
- PR creator must be user account (r.ciesielski3), no AI attribution in commit messages
- Never merge PR with unresolved comments
- Model assignment: Haiku for commits/notes, Sonnet for review decisions, Opus for final validation
- Current state: release-automation branch with 6 commits ahead of main, all fixes already applied
